# SIoT-HDT Opportunistic Sensing Simulator

This repository contains the synthetic SIoT-HDT simulator used to evaluate SIoT-enabled opportunistic sensing and source recruitment strategies for Human Digital Twin context estimation under partial observability.

The simulator is designed for controlled methodological comparison. It is not intended to predict deployment performance for a particular real system.

## Scientific Scope

- All data are synthetic.
- No real human-subject data are included.
- No personal, sensitive, or proprietary data are included.
- All SIoT graphs, source attributes, context states, observations, tables, and figures are generated from deterministic seeds.
- The simulator evaluates context-layer estimation under partial observability using transparent graph discovery, interpretable ranking, and weighted evidence fusion.

## Main Features

- Explicit Social Internet of Things graph layer.
- Typed SIoT relationships:
  - `OOR`: Ownership Object Relationship
  - `CLOR`: Co-Location Object Relationship
  - `CWOR`: Co-Work Object Relationship
  - `SOR`: Social Object Relationship
  - `POR`: Parental Object Relationship
- Graph-constrained candidate discovery.
- Exhaustive hop-limited discovery for high-coverage baselines.
- Bounded relationship-guided discovery for SIoT-aware selective policies.
- Four main recruitment policies:
  - `ego_only`
  - `opportunistic_all`
  - `siot_aware`
  - `siot_aware_trust_privacy`
- Budget-matched secondary baselines:
  - `random_k`
  - `budgeted_opportunistic_k`
- Cost-aware metrics:
  - `discovery_cost`
  - `recruitment_cost`
  - `selection_cost`
  - `total_operational_cost`
  - `effective_recovery_cost`
- Scalability analysis for SIoT graph sizes `N = 60, 200, 800`.
- Error analysis, sensitivity analysis, Pareto/cost-effectiveness analysis, and discovery-mode comparison.

## Discovery Modes

The simulator supports two SIoT discovery modes.

- `exhaustive_hop`: traverses all graph-reachable source candidates within the active hop radius. This mode is used by `opportunistic_all`, `random_k`, and `budgeted_opportunistic_k`, and is retained as a high-coverage/high-cost reference.
- `bounded_relationship_guided`: expands from the ego node with a deterministic priority order that favors shorter paths, stronger SIoT relations, higher expected relevance/trust/freshness, lower expected privacy and latency cost, and added coverage over unresolved context variables. This mode is used by `siot_aware` and `siot_aware_trust_privacy` by default.

Bounded discovery uses active node, edge, and scoring budgets plus a per-expansion neighbor beam. Default budgets are graph-size aware: `20/40/12` for small graphs, `30/70/18` for medium graphs, and `40/100/24` for large graphs, where the tuple is node budget / edge budget / max candidates to score.

Adaptive broadening increases the bounded discovery budgets, hop radius, and selective recruitment cap when ego evidence is missing, low-confidence, or unresolved. It does not switch SIoT-aware policies into exhaustive opportunistic behavior.

The discovery-mode comparison uses explicit variants:

- `siot_aware_exhaustive_discovery`
- `siot_aware_bounded_discovery`
- `siot_aware_trust_privacy_exhaustive_discovery`
- `siot_aware_trust_privacy_bounded_discovery`

Table 1 and the discovery-mode comparison are separate experiment outputs. Table 1 uses the `main_default` experiment tag with the default full run size. The discovery-mode comparison uses the `discovery_mode_comparison` experiment tag with explicit exhaustive/bounded policy variants and a compact deterministic run. Values from these files should be compared as separate experimental configurations unless their metadata columns match. The file `data/consistency_check_report.txt` records the current source rows and explains apparent Table 1 versus discovery-comparison differences.

## Context Variables

The simulator estimates four discrete context variables:

- `place`: `home`, `office`, `transit`, `meeting_room`, `public_place`
- `activity`: `focused_work`, `commuting`, `waiting`, `interacting`, `resting`
- `env_load`: `low`, `medium`, `high`
- `resource_state`: `nominal`, `degraded`, `sparse`

Context states are generated by an interpretable latent transition process. Place influences plausible activity, activity and environmental load influence resource state, and environmental/resource stress affects observation quality.

## Policy Labels

- `ego_only`: Ego-only baseline
- `opportunistic_all`: Opportunistic-all high-coverage baseline
- `siot_aware`: SIoT-aware bounded relationship-guided recruitment
- `siot_aware_trust_privacy`: Privacy-aware SIoT bounded relationship-guided recruitment
- `random_k`: Random-k budget-matched baseline
- `budgeted_opportunistic_k`: Budgeted opportunistic baseline

## Scenario Labels

- `nominal`: nominal conditions
- `degraded_ego`: degraded ego sensing
- `ambiguous_local_context`: ambiguous local context
- `noisy_untrusted_external`: noisy/untrusted external sources

## Reproducing the Results

Run the full deterministic experiment suite:

```bash
python run_experiments.py
```

The run regenerates:

- CSV outputs in `data/`
- PNG and PDF figures in `figures/`
- validation status in `data/validation_report.txt`

The step-level output `data/raw_step_results.csv` is generated locally by the simulator. Because the uncompressed file is larger than GitHub's normal file-size limit, the repository versions the compressed copy `data/raw_step_results.csv.gz`.

The default full run uses:

- episodes: `50`
- timesteps per episode: `48`
- seeds: `7, 11, 13, 17, 19`
- main graph size: `60` source nodes
- scalability graph sizes: `60, 200, 800`

Useful quick smoke run:

```bash
python run_experiments.py \
  --episodes 1 \
  --timesteps 6 \
  --seeds 7 \
  --budget-matched-episodes 1 \
  --budget-matched-timesteps 6 \
  --budget-matched-seeds 7 \
  --discovery-mode-comparison-episodes 1 \
  --discovery-mode-comparison-timesteps 6 \
  --discovery-mode-comparison-seeds 7 \
  --no-sensitivity \
  --no-scalability
```

## Validation

Each full run writes `data/validation_report.txt`. The validation checks:

- required CSV and figure files;
- graph topology and relation coverage;
- discovery-mode routing;
- bounded-discovery budgets;
- policy source-count rules;
- cost non-negativity;
- OCA/MVA/F1 ranges;
- seed recording;
- metadata columns in key CSV files;
- Table 1 consistency with `data/aggregated_metrics.csv`.

The expected final line is:

```text
Overall status: PASS
```
